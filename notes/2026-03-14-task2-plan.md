## Seven Issues to Address

**Issue 1: Self-Report Motivation Unmotivated**
- **Problem:** The paper opens with "models report highest confidence precisely when they are fabricating" (AUC 0.28–0.36 across four architectures).
- **Issue:** These numbers seem shocking, but they're asserted without evidence. Where's the actual data? Are self-report confidence and entropy measured on the same outputs?
- **Fix:** Either (a) add a side-by-side comparison of raw confidence distributions vs. entropy for a few example queries, or (b) reference a prior work that shows this pattern, or (c) explicitly note "see Section 4 for raw data" and add a figure showing the confidence inversion.
- **Why it matters:** This is the motivation for the entire paper. If readers think "this might be measurement artifact," the whole edifice wobbles.

**Issue 2: Length Control Methodology Vague**
- **Problem:** Section 4 says "under length-controlled evaluation where we control for this asymmetry, AUC dropped to 0.63."
- **Issue:** What does "control for" mean precisely? Did you match length distributions between knowable/unknowable queries? Hold length constant? Residualize entropy on length?
- **Fix:** Add 1-2 sentences explaining the exact procedure. The Appendix Detailed Methodology section (lines 1471–1485) computes budget curves but doesn't explain length control itself.
- **Why it matters:** Reviewers will ask this. Vagueness looks like you're hiding something.

**Issue 3: Tensor Signal Aggregation Unclear**
- **Problem:** Line 598 claims "+2.4% overhead for entropy, +7.1% for full signal set." But what's in each?
- **Issue:** "Entropy" is clear. But "full signal set" = entropy + attention summaries + log-probabilities. How are attention summaries aggregated? (TDA mentions Jaccard index, layer-wise head concentration—but is that in the main signal or appendix-only?)
- **Fix:** Add one paragraph in Section 3 (Design) explicitly defining:
  - Per-token entropy: [definition]
  - Attention summaries: [which statistics: mean? variance? topological coherence?]
  - Log-probabilities: [per-token or aggregate?]
  - Overhead: [clarify what's measured]
- **Why it matters:** "Full signal set" is vague. Make it concrete.

**Issue 4: Citation Inversion Lift is Modest**
- **Problem:** Composed judge (tensor + bounded lookup) gets 91.8% vs. tensor-only 90.2% at 30% budget. That's +1.6pp.
- **Issue:** The paper spends lines 841–851 motivating composition by the citation inversion insight (fabricated citations = low entropy because fluent, real citations = high entropy because retrieval). But the empirical lift is tiny.
- **Fix:** Either (a) show that composition helps MORE on citation-heavy datasets, or (b) frame composition as "handles domain-specific failure modes" not "major improvement." The current framing oversells the benefit.
- **Why it matters:** Reviewers will note the gap between theory (elegant insight) and practice (tiny lift).

**Issue 5: RAG Escape Scope is Narrow**
- **Problem:** Theorem 2 (Learnability) introduces the retrieved documents case (RAG). But it relies on the Hallucination Regime condition: there exists a plausible fabrication r_fab whose verification cost exceeds budget B.
- **Issue:** In an ideal RAG with perfect retrieval and perfect verification, this might not hold. The paper acknowledges (lines 464–471) that scope is "Deployment-Time Supervision" (inference-time supervisors with bounded budget), not training-time. But this scope limitation might deserve more prominence.
- **Fix:** Add one sentence: "This result applies to deployment-time supervisors operating under budget constraints. Training-time supervision with unlimited budget is outside our scope." (This is true but could be underlined.)
- **Why it matters:** Reviewer might ask "but what if we train the system to be honest?" Answer: that's a different problem, outside this paper's scope.

**Issue 6: Responsibility Concentration Underdeveloped**
- **Problem:** The paper has a Corollary (line 560–571) stating "when the provider retains exclusive access to epistemic telemetry, only the provider can verify epistemic honesty."
- **Issue:** This is a huge governance implication—it determines who bears the cost of verification. But it's relegated to a remark. It deserves a section.
- **Fix:** Expand this into a short subsection (3–4 paragraphs) in Section 5 (Discussion) under "Policy Implications" or "Governance." Explain:
  - If provider exports entropy: external verification is possible
  - If provider doesn't: only provider can verify
  - Implications for accountability, liability, auditability
- **Why it matters:** This is where the research hits governance. Don't hide it.

**Issue 7: Ground Truth is Synthetic, Real Deployments May Differ**
- **Problem:** The evaluation uses a balanced 200-query set (100 knowable, 100 unknowable). This is controlled and clean.
- **Issue:** Real deployments have unknown ratios of knowable:unknowable, and the types of unknowns vary wildly. The cost surface (Figure at line 889–894) is calibrated to this synthetic world.
- **Fix:** Add a brief note in Section 5 (Limitations): "The cost surface is calibrated to a balanced query set. Real deployments will have different base rates and query distributions. Systems must recalibrate entropy thresholds on domain-specific data."
- **Why it matters:** Practitioners need to know: this table applies to balanced data, not real data. Calibration is required.

**Bonus Issue (not critical but worth considering):**
- **Failure cases not shown.** Which queries does entropy get wrong? Show a ranked list (top 10 misclassified). This grounds the cost surface and makes the remaining 8.3% error rate feel real.
